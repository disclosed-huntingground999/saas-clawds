# 23 — Documentation: Knowledge That Scales

> Documentation is the bridge between your product and your users (and your team). This department covers user docs, API docs, changelog, internal docs, and developer guides — so everyone can find what they need.

## Purpose

Good documentation reduces support load, speeds up onboarding, enables developers to integrate, and preserves institutional knowledge. This department structures your documentation strategy across user-facing, API, and internal docs.

| Subfolder | What You'll Produce | Priority |
|---|---|---|
| [user-docs](./user-docs/) | Help center structure, getting started, feature docs | P0 |
| [api-docs](./api-docs/) | API reference, auth guide, code examples | P1 |
| [changelog](./changelog/) | Release notes format, versioning, notifications | P1 |
| [internal-docs](./internal-docs/) | Runbooks, architecture, ADRs | P1 |
| [developer-guides](./developer-guides/) | Integration tutorials, webhooks, dev portal | P2 |

## How to Work Through This Department

1. **User Docs** — Help center, getting started, feature documentation
2. **API Docs** — API reference (if applicable), auth, examples
3. **Changelog** — Release notes format and distribution
4. **Internal Docs** — Runbooks, architecture, decision records
5. **Developer Guides** — Integration tutorials, webhooks, developer portal

## Time Estimate

| Approach | Time |
|---|---|
| User docs + changelog | 1–2 weeks |
| + API docs + internal | 2–3 weeks |

## Key Principle

**Document for the reader.** User docs = what can I do? API docs = how do I call this? Internal docs = how does this work?
